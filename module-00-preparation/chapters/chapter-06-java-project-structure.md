# Chapter 6 — Java Project Structure

## Overview

A well-organized project is one of the foundations of professional software development.

As Java applications grow, maintaining a clean and consistent structure becomes essential for readability, scalability, and collaboration.

This chapter explains how Java projects are organized, the purpose of each directory, and how packages help keep code clean, modular, and maintainable.

By following standard project structures and naming conventions, developers can build applications that are easier to understand, extend, and collaborate on.

- How a Java project is structured
- The purpose of each project directory
- What packages are and why they exist

- The importance of separating responsibilities

## Typical Java Project Structure

A Java project is organized into directories and files, each with a specific purpose.

A typical project may look like this:

```text
MyJavaProject/
│
├── src/
├── out/
├── README.md
├── .gitignore
└── pom.xml (optional)
```

Each element contributes to the organization and maintainability of the application.

---

## The `src` Directory

The `src` (source) directory contains all the Java source code written by the developer.

Example:

```text
src/
└── com/
    └── eduardohenrique/
        └── javabootcamp/
            └── Main.java
```

Everything inside this directory is compiled into bytecode during the build process.

The `src` directory is considered the heart of every Java project because it contains the application's logic.

---

## The `out` Directory

The `out` directory contains compiled files generated by the IDE or the Java compiler.

Example:

```text
out/
└── production/
    └── MyJavaProject/
        └── Main.class
```

This directory is automatically generated and usually contains:

- Compiled `.class` files
- Temporary build artifacts

Since these files can always be recreated, they should not be edited manually or committed to Git.

---

## The `README.md` File

The `README.md` file is the project's main documentation.

A good README should explain:

- The project's purpose
- Technologies used
- Folder structure
- Installation instructions
- How to run the application
- Learning objectives (for educational projects)

Well-written documentation makes a project easier to understand for both yourself and other developers.

---

## The `.gitignore` File

The `.gitignore` file tells Git which files and directories should not be tracked.

Common entries include:

```text
out/
.idea/
*.class
```

Ignoring generated files keeps the repository clean and prevents unnecessary files from being committed.

---

## Understanding Packages

Packages are used to organize Java classes into logical groups.

Instead of placing every class in the same folder, related classes are grouped together according to their responsibilities.

Example:

```text
com.eduardohenrique.javabootcamp
```

Directory structure:

```text
com/
└── eduardohenrique/
    └── javabootcamp/
```

Each package directly corresponds to a folder structure.

---

## Why Packages Exist

Packages solve several important problems in software development.

They help:

- Organize related classes
- Avoid class name conflicts
- Improve readability
- Simplify maintenance
- Support modular application design
- Make navigation easier in large projects

Without packages, managing large applications would become extremely difficult.

---

## Package Naming Convention

Java follows official naming conventions for packages.

Rules:

- Use only lowercase letters.
- Separate package levels using dots (`.`).
- Begin with the reversed internet domain when applicable.

Examples:

```text
com.company.project
```

```text
org.springframework.boot
```

Personal projects can follow the same convention:

```text
com.eduardohenrique.javabootcamp
```

Following these conventions ensures consistency and avoids naming conflicts.

---

## Organizing Responsibilities

As projects grow, classes should be separated by their responsibilities.

Example:

```text
com.eduardohenrique.javabootcamp
│
├── app
├── model
├── service
├── repository
├── controller
└── util
```

Each package serves a different purpose:

| Package    | Responsibility          |
| ---------- | ----------------------- |
| app        | Application entry point |
| model      | Business entities       |
| service    | Business logic          |
| repository | Data access             |
| controller | Request handling        |
| util       | Utility classes         |

Separating responsibilities makes applications easier to maintain, test, and extend.

---

## Best Practices

- Organize projects from the beginning.
- Use meaningful package names.
- Keep related classes together.
- Follow Java naming conventions.
- Separate responsibilities into dedicated packages.
- Keep the project structure simple and consistent.

---

## Common Mistakes

- Using the default package.
- Mixing unrelated classes in the same package.
- Creating unnecessary package levels.
- Using uppercase letters in package names.
- Committing generated files (`out`, `.class`, `.idea`) to Git.

---

## Key Concepts Summary

- Java projects follow a standardized directory structure.
- The `src` directory stores source code.
- The `out` directory stores compiled files.
- Packages organize related classes.
- Package names should always be lowercase.
- Good organization improves scalability and maintainability.
